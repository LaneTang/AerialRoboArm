# Runtime Architecture v1.0

## 1. 文档定位
本文件用于定义 ARA 的运行时架构，即系统在上电后如何按既定节拍组织任务执行、如何在不同频率域之间传递信息、如何划分机载决策路径与执行路径，以及如何在不破坏资源边界与时序确定性的前提下支撑 Manual / Auto 两类主业务流。

本文件描述的是**系统应该如何运行**，而不是某一版本中具体代码已经实现到哪一步。线程周期数值、桥接方式、数据流方向、责任边界等属于本文件约束范围；模块完成度、联调进展、暂存 workaround 与具体调参结果不属于本文件约束范围。

## 2. 运行时架构总览
ARA 运行时架构采用“**双物理线程容器 + 多 Runnable 管线 + 统一跨线程数据桥**”的总体形式。

其核心思想是：
- 用极少数、职责固定的物理线程承载系统运行时节拍；
- 将具体业务收敛为无阻塞 Runnable，由调度容器按固定周期调用；
- 将高频控制域与低频业务域之间的交互，收敛到统一的跨线程桥梁；
- 保持从输入语义到动作输出再到状态反馈的单向数据流；
- 在资源受限 MCU 上，以确定性优先于表面上的“灵活多任务”。

## 3. 软件架构分层与运行时角色

### 3.1 L1：BSP 层
L1 负责芯片级外设抽象，提供 GPIO、PWM、UART、I2C 等基础硬件接口。L1 不承担任何业务语义，不直接表达系统模式与控制意图。

### 3.2 L2：Driver 层
L2 负责将具体硬件设备封装为可复用、无业务语义的设备驱动，例如编码器、BLDC 功率级、ELRS 链路、舵机等。L2 不直接决定系统行为。

### 3.3 L3：Algorithm / Module 层
L3 负责算法与功能模块封装，例如 FOC、PID、RC 语义映射、末端执行器抽象、VSP 解析等。L3 允许表达局部功能语义，但不承担系统运行调度职责。

### 3.4 L4：Task Runnable / FSM 层
L4 负责将具体业务组织成按固定节拍调用的 Runnable，并在该层承载系统模式逻辑与业务状态机。L4 不拥有物理线程，不直接创建任务容器，而是被 L5 周期性调用。

### 3.5 L5：App Threads / Scheduler 层
L5 是运行时架构的物理承载层。L5 负责建立固定节拍线程容器、按节拍调用各 Runnable、组织跨线程桥接与系统级调度。L5 是系统运行时的唯一上位时序组织者。fileciteturn1file0

## 4. 双物理线程容器

### 4.1 架构原则
系统采用两个物理线程容器：
- 高频控制线程
- 低频逻辑线程

之所以采用双物理线程，而不是按业务拆分多个 RTOS 任务，是因为本系统的核心矛盾不在“并发业务种类多”，而在“高频控制链路必须稳定、RAM 极其有限、线程切换与栈开销不可滥用”。双物理线程架构的目标，是以最小运行时开销换取最强时序可预测性。

### 4.2 高频控制线程
高频线程运行在 1ms 周期，用于承载动力核心相关的高频控制逻辑。当前阶段，高频线程唯一应承载的核心 Runnable 是 Motion 控制链路。其职责包括：
- 读取上一周期有效的主轴状态反馈；
- 执行主轴运动状态机的高频部分；
- 运行位置闭环与 FOC 输出；
- 更新三相 PWM；
- 触发下一次传感器更新。

高频线程不得承担复杂业务判断、字符串格式化、阻塞式外设访问、低优先级状态打印或与视觉语义直接相关的慢路径逻辑。fileciteturn1file19

### 4.3 低频逻辑线程
低频线程运行在 20ms 周期，用于承载业务逻辑、输入语义处理、状态机决策与系统观测输出。其职责包括：
- 处理 ELRS / RC 输入并生成人工控制语义；
- 处理 Mock Vision 或真实视觉语义输入；
- 执行 Manipulator 业务状态机；
- 组织 Manual / Auto 模式下的动作目标生成；
- 汇总低频状态输出与观测信息；
- 负责与高频线程之间的命令 / 状态桥接。

低频线程允许承担业务决策与控制主权分配，但不得直接承担高频闭环执行。fileciteturn1file20 fileciteturn1file21 fileciteturn1file22

## 5. 频率域划分原则
系统运行时被划分为两个频率域：

### 5.1 高频控制域
高频控制域服务于执行器闭环控制，强调：
- 固定周期
- 极短执行时间
- 无阻塞
- 低分支复杂度
- 高可预测性

当前版本中，只有 BLDC 主轴属于高频闭环对象，因此高频控制域应围绕主轴动力链路组织。

### 5.2 低频业务域
低频业务域服务于输入解释、状态机推进、目标生成与系统观测。该域强调：
- 语义解释能力
- 模式切换
- 动作序列组织
- 外部输入适配
- 联调与观测友好性

视觉事件、RC 输入、Manipulator 状态机与 Servo 业务控制均应归入低频业务域。

## 6. 跨线程数据桥：DataHub
DataHub 是当前架构中高频控制域与低频业务域之间的唯一正式跨线程桥梁。其职责不是承载复杂业务逻辑，而是以紧凑、小结构体、低开销的方式，在两个频率域之间传递最必要的命令与反馈。fileciteturn1file11

### 6.1 下行命令桥
低频逻辑线程根据当前模式与业务状态机生成 `DataHub_Cmd`，下发至高频控制线程。其本质是“低频决策结果”向“高频执行核心”的唯一动作输入。

### 6.2 上行状态桥
高频控制线程将主轴当前状态写入 `DataHub_State`，供低频线程做模式判断、业务观测与必要的决策参考。其本质是“高频执行结果”向“低频业务域”的唯一反馈输出。

### 6.3 桥接设计原则
DataHub 必须服从以下原则：
- 只承载小而稳定的结构体；
- 不承载大对象与原始感知数据流；
- 不引入复杂逻辑；
- 不作为日志缓存池；
- 不作为模块间任意共享变量的替代品；
- 其存在意义是桥接，而不是扩张为“万能全局黑板”。

## 7. 输入路径架构

### 7.1 RC 输入路径
RC 输入路径负责将 ELRS 原始通道数据转化为系统能够理解的人工控制语义。该路径的逻辑方向为：

ELRS 链路 -> 原始通道解析 -> RC 语义映射 -> 模式请求 / 主轴动作意图 / 夹爪动作意图 / 姿态控制输入

RC 输入路径是系统的人工接管入口。在运行时架构上，它属于低频业务域。

### 7.2 视觉输入路径
视觉输入路径负责将统一扩展链路输入的最终语义事件提供给机载业务状态机。该路径不关心 ToF、单目或其融合算法的内部实现，而只关心最后进入控制链路的语义结果。

在运行时架构上，视觉输入路径也属于低频业务域。

### 7.3 Mock Vision 与真实视觉链路
为支持开发期联调，系统允许在运行时架构中保留 Mock Vision 与真实视觉输入的替换关系。二者在进入业务状态机之前，应尽可能被统一为同一类视觉语义输入，以避免上层业务逻辑因为输入来源不同而分裂。fileciteturn1file21

## 8. 输出路径架构

### 8.1 BLDC 主轴输出路径
BLDC 主轴输出路径是系统当前唯一的高频动力闭环路径，其方向为：

低频目标生成 -> DataHub 下行命令 -> Motion 高频控制 -> FOC / PWM -> BLDC 主轴

该路径是系统的动力核心。

### 8.2 Servo 输出路径
Servo 输出路径承担夹爪与末端姿态控制，其本质是低频语义驱动的执行器输出路径。其方向为：

低频业务状态机 / RC 语义 -> Servo 功能模块 -> PWM 输出 -> 夹爪 / 姿态执行器

该路径不进入当前版本的统一高频闭环架构。

## 9. Manual / Auto 两条主业务流

### 9.1 Manual Path
在 MANUAL 模式下：
- RC 输入成为唯一动作主导来源；
- 自动决策逻辑挂起；
- 视觉语义允许继续更新，但不主导动作输出；
- 主轴、夹爪与姿态执行器依据 RC 语义进行响应。

Manual Path 的目标是确保人工接管路径清晰、直接、可解释。

### 9.2 Auto Path
在 AUTO 模式下：
- 系统接收来自统一扩展链路的最终视觉语义事件；
- Manipulator 业务状态机根据视觉事件与当前系统状态生成动作目标；
- 高频 Motion 路径执行主轴闭环；
- Servo 路径执行姿态与夹取动作；
- 系统形成“感知语义 -> 机载决策 -> 执行动作 -> 主轴反馈”的基本闭环。

Auto Path 的目标是实现受监督的局部自治，而非完全独立于人工接管能力的全自主系统。

## 10. Happy Path 运行时表达
当前阶段的标准 Happy Path 可概括为：

1. 机载感知链路持续采集前方环境信息；
2. 地面端或感知端将结果压缩为最终视觉语义事件；
3. 视觉语义事件通过统一扩展链路送入 ARA 系统；
4. 低频业务状态机根据事件判断目标是否存在、是否需要姿态调整、是否进入抓取流程；
5. 高频 Motion 路径驱动 BLDC 主轴执行接近、展开或收回动作；
6. Servo 路径执行末端姿态与夹爪动作；
7. 主轴状态通过反馈路径回流至低频业务域，支持后续业务推进。

Happy Path 是当前运行时架构首要保障的标准业务流，其成立优先级高于复杂异常分支的完备覆盖。

## 11. 启动与运行时阶段划分
从运行时角度，系统应被理解为依次经历以下阶段：

### 11.1 上电准备阶段
完成 BSP、Driver、DataHub 与 L4 / L5 上下文初始化。

### 11.2 动力链路建立阶段
完成主轴动力核心所需的对齐、参考建立与初始可控状态确认。

### 11.3 系统待命阶段
系统已具备基本可控性，等待进入 MANUAL 或 AUTO 主业务流。

### 11.4 主业务运行阶段
系统依据当前模式进入人工控制或局部自治控制。

这一阶段划分用于解释运行时结构，不等同于具体代码中的枚举定义或某一实现版本的状态机细节。

## 12. 运行时约束
运行时架构必须长期服从以下约束：

### 12.1 不得用新增线程替代架构问题
除非资源与时序分析明确证明必要性，否则不得以“新增独立线程”作为默认解法来处理新功能接入。

### 12.2 不得在高频路径中引入慢操作
任何可能导致高频控制路径抖动的行为，例如阻塞式外设访问、复杂格式化日志、长耗时分支、不可控等待，均不得进入高频线程。

### 12.3 不得让输入源直接绕过业务层支配执行器
无论是视觉输入还是调试链路输入，都不应绕过低频业务层直接操控主闭环执行路径。

### 12.4 不得让模块横向偷读内部状态
模块之间若需共享正式业务信息，应通过已定义的桥梁与接口完成，而不是通过临时 extern、全局变量或上下文直接穿透实现。
