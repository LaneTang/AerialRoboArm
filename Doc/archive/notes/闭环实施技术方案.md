# AerialRoboArm 闭环实施技术方案

## 1. 文档目的

本文档用于定义 AerialRoboArm 当前阶段的正式闭环实施路线，明确：

- 当前项目的真实代码现状
- 闭环控制目标与边界
- 推荐的软件架构与任务分工
- 分阶段实施步骤
- 关键风险点与验证方法

本文档面向：

- Designer：用于维护顶层控制架构
- APIer：用于固化模块接口与上下层依赖
- Coder：用于按阶段实现与调试闭环链路

---

## 2. 当前项目现状审查

基于当前仓库代码审查，项目中已经存在一条闭环控制雏形，但尚未作为主运行路径投入使用。

### 2.1 当前可确认的能力

当前仓库已经具备以下基础模块：

- 编码器驱动：`User/drv/Src/drv_as5600.c`
- 三相驱动控制：`User/drv/Src/drv_bldc_power.c`
- PWM 输出封装：`User/bsp/Src/bsp_pwm.c`
- I2C DMA 读取：`User/bsp/Src/bsp_i2c.c`
- FOC 算法核心：`User/mod/Src/alg_voltage_foc.c`
- PID 控制器：`User/mod/Src/alg_pid.c`
- 闭环任务雏形：`User/task/Src/task_motion.c`

### 2.2 当前实际主运行链

当前实际主要运行的是 RC 联调链，而非正式闭环链：

- `ELRS -> RC Semantic -> test_rc_console -> m mode`

其中 `m mode` 当前主要用于：

- 联调遥控输入
- 验证急停与模式切换
- 验证电机输出链是否打通

注意：

- 当前 `m mode` 不是正式闭环控制
- 它属于联调/保底测试路径

### 2.3 当前闭环链未真正接入的原因

当前闭环主任务 `TaskMotion` 尚未真正作为系统主控制链运行，主要表现为：

- `TaskMotion_Init()` 中任务创建被注释
- `freertos.c` 中 `MotorTask()` 为空循环
- `App_Scheduler_Entry()` 未真正接入当前任务主线

### 2.4 当前主要风险

当前闭环实施前的主要风险有：

1. 编码器链路稳定性尚未完全验证
2. I2C DMA 偶发 BUSY/超时历史问题仍需重点关注
3. 当前速度估计为原始差分，缺少滤波
4. FOC 零点对齐、极对数和方向关系尚未形成完整验证闭环

---

## 3. 闭环实施目标

本阶段闭环实施不以“一步到位的位置闭环机械臂控制”为目标，而采用逐层收敛策略。

### 3.1 总目标

建立一条可稳定运行的正式闭环控制链，使系统具备：

- 编码器角度反馈
- 电机速度闭环控制
- 安全停机与异常退出机制
- 后续扩展位置闭环的架构基础

### 3.2 分阶段目标

#### 阶段 V1：最小速度闭环

目标：

- 基于 AS5600 角度反馈
- 建立单环速度闭环
- 实现低速、稳定、可重复控制

#### 阶段 V2：遥控速度闭环

目标：

- 将 ELRS/上位机目标映射到目标速度
- 保持安全逻辑（EStop / ModeReq）
- 用闭环代替当前 `m mode` 的开环联调输出

#### 阶段 V3：位置-速度双环

目标：

- 外环位置 PID
- 内环速度 PID
- 实现面向机械臂动作的目标角度控制

---

## 4. 推荐闭环架构

### 4.1 总体链路

推荐正式闭环链路如下：

`目标输入 -> TaskMotion -> AS5600反馈 -> 速度估计 -> 速度PID -> FOC -> BLDC驱动 -> 电机`

在后续位置环接入后，扩展为：

`目标位置 -> 位置PID -> 目标速度 -> 速度PID -> FOC -> BLDC驱动`

### 4.2 软件分层建议

#### L1 BSP 层

职责：

- PWM 输出
- I2C DMA
- GPIO 使能
- UART 调试

对应文件：

- `User/bsp/Src/bsp_pwm.c`
- `User/bsp/Src/bsp_i2c.c`
- `User/bsp/Src/bsp_gpio.c`
- `User/bsp/Src/bsp_uart.c`

#### L2 驱动层

职责：

- AS5600 设备抽象
- BLDC 功率级抽象

对应文件：

- `User/drv/Src/drv_as5600.c`
- `User/drv/Src/drv_bldc_power.c`

#### L3 算法/语义层

职责：

- FOC 算法
- PID 算法
- 遥控语义转换

对应文件：

- `User/mod/Src/alg_voltage_foc.c`
- `User/mod/Src/alg_pid.c`
- `User/mod/Src/mod_rc_semantic.c`

#### L4 任务层

职责：

- 闭环控制任务
- 遥控任务
- DataHub 数据交换

对应文件：

- `User/task/Src/task_motion.c`
- `User/task/Src/task_rc.c`
- `User/global/src/datahub.c`

#### L5 调度/应用层

职责：

- 模式切换
- 错误态仲裁
- 系统状态机

对应文件：

- `User/app/Src/app_scheduler.c`

### 4.3 运行时任务分工建议

建议将正式运行任务职责定义为：

- `LogTask`
  - 负责串口日志与测试台
  - 不直接控制电机

- `MotorTask`
  - 负责调用 `TaskMotion_Entry()`
  - 作为正式闭环控制主任务

- `RC Task / DefaultTask`
  - 负责读取遥控输入
  - 生成目标速度或目标位置
  - 不直接输出三相 PWM

## 5. 闭环控制策略

## 5.1 阶段 V1：速度闭环

### 目标

先实现最小可运行速度闭环，不直接上位置环。

### 原因

这是当前项目最稳妥的技术路线，原因如下：

- 当前位置环会叠加更多不确定性
- 当前速度估计与编码器链路还需要稳定验证
- 先打通速度闭环更容易定位问题

### 控制流程

1. 读取编码器角度 `raw_angle`
2. 使用角度差分估计当前速度
3. 使用速度 PID 计算 `u_q`
4. 使用 FOC 算法计算三相 duty
5. 更新 PWM 输出

### 当前代码基础

该流程在 `User/task/Src/task_motion.c` 中已基本具备。

### 建议优化

在正式投入运行前，建议增加：

- 简单速度低通滤波
- 对速度单位进行统一定义
- 对零点校准结果增加日志输出

## 5.2 阶段 V2：遥控速度闭环

### 目标

将遥控命令由“直接驱动开环动作”切换为“设置闭环目标速度”。

### 推荐语义映射

- `EStop`
  - 最高优先级
  - 激活时立即关闭驱动输出

- `ModeReq`
  - 决定是否允许人工接管

- `ArmCmd`
  - 不再直接下发三相换相
  - 改为：
    - `EXTEND -> 正向目标速度`
    - `RETRACT -> 反向目标速度`
    - `HOLD -> 目标速度 = 0`

### 目标速度建议

初始调试阶段建议使用固定离散目标值，而非直接将摇杆映射为连续量。

例如：

- `EXTEND -> +V_test`
- `RETRACT -> -V_test`
- `HOLD -> 0`

这样更容易判断闭环是否稳定。

## 5.3 阶段 V3：位置-速度双环

### 目标

在速度闭环稳定的前提下，叠加位置外环。

### 控制流程

1. 目标位置 - 当前角度 = 位置误差
2. 位置 PID 输出目标速度
3. 速度 PID 输出目标转矩/电压指令
4. FOC 输出三相 PWM

### 适用场景

- 机械臂目标姿态控制
- 指定角度控制
- 平滑点到点运动

## 6. 关键实现步骤

## 步骤 1：正式接入 TaskMotion

### 目标

让 `TaskMotion` 真正运行起来，代替当前仅用于联调的开环测试路径。

### 建议做法

在 `Core/Src/freertos.c` 中：

- 在 `MotorTask()` 中调用 `TaskMotion_Init()`
- 然后直接进入 `TaskMotion_Entry()`

不建议继续依赖 `TaskMotion_Init()` 内部 `xTaskCreate()` 方式创建任务，推荐统一由 FreeRTOS 线程入口管理。

## 步骤 2：固定目标速度闭环实验

### 目标

先在不接入 ELRS 的前提下验证闭环。

### 建议

在 `TaskMotion` 中临时固定一个较小的目标速度：

- 正向低速
- 空载测试
- 不挂机械臂负载

通过此步骤验证：

- 编码器是否稳定读数
- 闭环是否能持续转动
- PID 是否有基本调节效果

## 步骤 3：编码器链稳定性验证

### 重点检查

- `DrvAS5600_Init()` 是否稳定通过
- `DrvAS5600_TriggerUpdate()` 是否长期稳定返回 `ARA_OK`
- `xSemaphoreTake()` 是否不出现频繁超时
- `raw_angle` 是否连续变化

### 若不稳定

优先处理 I2C 链路，暂不继续推进闭环调参。

## 步骤 4：速度估计滤波

### 当前问题

当前速度估计为简单差分：

- 噪声敏感
- 对 PID 不友好

### 建议

先加入最简单的一阶低通滤波：

- 不改变整体架构
- 先降低测量抖动

## 步骤 5：速度 PID 调参

### 建议策略

先调 `P`，再加 `I`，最后视情况是否使用 `D`。

推荐起步策略：

- `Kp` 小值起步
- `Ki` 极小
- `Kd = 0`

### 原因

当前速度信号仍偏原始，先避免 D 项放大噪声。

## 步骤 6：接入遥控闭环

### 目标

让 ELRS 只负责提供目标，不再直接下发开环三相动作。

### 推荐改法

保留 `m mode` 作为联调和保底测试工具，但新增正式闭环模式：

- 遥控输入 -> 目标速度
- `TaskMotion` 执行闭环控制

## 7. 参数建议

以下参数为初期调试建议，不作为最终定值。

### 控制周期

- 优先目标：`1ms`
- 若 I2C 不稳定：可临时退让到 `2ms ~ 5ms`

### 对齐阶段

当前值：

- `ALIGN_DURATION_TICKS = 2000`
- `ALIGN_VOLTAGE_Q15 = 3276`

建议：

- 保留当前对齐策略
- 增加对 `zero_offset` 的串口输出

### PID

当前代码起始值：

- `Kp = 200`
- `Ki = 10`
- `Kd = 0`

建议：

- 初始仅作保守启动值
- 必须基于实际闭环表现再调

## 8. 风险清单

### 风险 1：AS5600 读取不稳定

表现：

- DMA 启动失败
- BUSY 错误
- 超时

影响：

- 闭环主链直接失效

优先级：

- 最高

### 风险 2：速度估计噪声过大

表现：

- 电机抖动
- PID 难调
- 容易振荡

优先级：

- 高

### 风险 3：零点校准或极对数不匹配

表现：

- 电机有力但不顺
- 转矩异常
- 闭环方向错误

优先级：

- 高

### 风险 4：FOC 算法实现细节未完全验证

表现：

- duty 计算异常
- 某些角度段运行不稳定

优先级：

- 中高

### 风险 5：串口日志影响实时性

表现：

- 控制周期抖动
- 测试现象与日志不同步

优先级：

- 中

## 9. 验证方案

## 9.1 V1 验证：固定速度闭环

验证项：

- 上电后能完成对齐
- 零点采样成功
- 电机可持续低速稳定旋转
- 停止目标速度后可平稳停下

## 9.2 V2 验证：遥控速度闭环

验证项：

- `EStop` 能立即切断
- `ModeReq` 能正确门控人工控制
- `EXTEND/RETRACT/HOLD` 可正确映射到目标速度

## 9.3 V3 验证：位置闭环

验证项：

- 指定目标角度可到达
- 无明显持续振荡
- 来回切换时稳定

---

## 10. 与当前开环测试路径的关系

当前 `test_rc_console.c` 中的 `m mode` 建议保留，但应明确其定位：

- 仅用于联调
- 用于验证输入链与功率级输出链
- 不作为正式闭环运行模式

也就是说：

- `m mode` 是保底工具
- `TaskMotion` 才是正式闭环主链

---

## 11. 结论

本项目当前最合理的闭环实施路线为：

1. 启用 `TaskMotion` 正式任务
2. 优先实现最小速度闭环
3. 稳定编码器与速度估计
4. 完成 PID 基本调参
5. 将遥控输入由“直接开环驱动”切换为“闭环目标输入”
6. 最后再扩展位置-速度双环

该方案的核心原则是：

- 先打通单环
- 再接入遥控
- 最后扩展双环

避免在当前阶段同时处理过多耦合问题。
